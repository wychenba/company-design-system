**Q1 Ant Design 深研**
Verdict: Round 1「Ant Select 用 source order」是錯的。Ant `Select` 本身沒有內建 Select All；官方 multiple / big-data demo 只示範 `mode="multiple"`、`defaultValue`、`options`、virtual scroll，沒有 select-all ordering contract。見 `components/select/demo/multiple.tsx`：`mode="multiple"`、`defaultValue={['a10', 'c12']}`、`options={options}`；big-data doc 只講 virtual scroll。

Ant 有內建 Select All 語意的是 Transfer / Table 類元件。Transfer list checkbox 用 `filteredItems.filter(...).map(({ key }) => key)`，也就是目前列表順序；dropdown select-all 用 `getEnabledItemKeys(filteredItems)`。但 Transfer state merge 是 `Array.from(new Set([...prevKeys, ...keys]))`，即「既有選取順序保留，bulk keys append」。這比較接近路徑 A，不是 B/C。

Ant Table filter API 層是 `filters` + `filterMultiple` + `FilterValue` array；公開型別有 `filterMultiple?: boolean`、`filteredValue?: FilterValue | null`。我沒有成功抓到目前 master 的 `FilterDropdown.tsx` 原始碼行號，GitHub/UNPKG 對該檔 fetch 不穩；所以 Table filter ordering 只能保守判成「filter keys array driven」，不能當成強 source-order 證據。GitHub issues 搜到的 Ant select-all 討論主要是 Table pagination scope，不是 selected chip ordering：#25747 明確討論「Select All checkbox selects current page」與 whole dataset 狀態。

| 系統 | source cite | ordering 行為 / 共識 |
|---|---|---|
| MUI Autocomplete | `multiple` 只保證 value 是 array；select-all feature request #21211 仍是 custom ListBox，且要求 select-all 固定在 top、非 options | 無內建 Select All；ordering 留給 controlled `value`。不能作為 source-order 共識。 |
| Shopify Polaris | `Select` 是單選，文件建議多選/進階格式用 popover + option list；ChoiceList `selected string[]` + `allowMultiple` | 沒有可引用的 SelectPanel built-in select-all ordering。Polaris 指南更重視 logical/alphabetical option order。 |
| Atlassian Forge UI Kit Select | Forge `Select` supports multi selection；community 明確問過 multi select select-all，答案是自己做 / react-select | Forge 無內建 select-all。另有排序爭議：Forge UI Kit auto-arranges into field stored order，被指出不同於 Atlassian Design example 的 user selection order。世界級沒有一致共識。 |

**Q3 α+γ Hybrid Verify**
Approve hybrid。

α 已是 slot-based：`slots = 1 + floor((available - avatar) / step)`，`total <= slots` 全顯，否則 `slots - 1` avatar + chip。這把 `+N` 當 stack 內最後一個圓，不是外加 chunk。

γ 也真的做 chip overlap：`OverflowTagList` 對 chip wrapper merge `overflowWrapperClassName`，PeoplePicker stack 傳 `-ml-0.5 first:ml-0 relative inline-flex`。

α+γ 視覺結果等價於 pure γ 的目標：avatar 和 chip 同尺寸、同 overlap step。差異只在實作層：Combobox 仍保留 measurement wrapper / hidden DOM contract；但 layout 物理不再是 chip-as-separate-chunk。Layer C screenshot / pixel verify 仍需 Layer A 跑，這個 sandbox 不能跑 Playwright。

**Q4 DS-Wide Audit Gap**
`MultiPersonDisplay` readonly path 已共用 formula primitive：measured path 呼叫 `getAvatarStackVisibleCount`；readonly visible avatars 用 `i > 0 ? '-ml-0.5'`，overflow chip 也有 `-ml-0.5`。

text-tag Combobox 沒被破壞：`overflowWrapperClassName` default 是 undefined；prop 註解明確說 default undefined = text-tag mode chip 不 overlap。grep 只有 PeoplePicker stack 傳這個 prop；DataTable multi-select text tags仍是普通 `<Combobox>`，沒有 opt-in overlap。

目前 drift 點：`Combobox` display/readonly text-tag path仍用 `ComboboxTagStack`，不是 PeoplePicker avatar formula；這是不同 visual idiom，合理。formula primitive 只被 `person-display.tsx` 和 `people-picker.tsx` 消費，沒有第三處 drift。

**Q5 Improvement Direction**
A. Preserve click order + append unselected at end  
Pros: 最接近 Ant Transfer merge：`[...prevKeys, ...keys]`。Cons: 對 filter / PeoplePicker chip display 會保留歷史操作痕跡，Select All 後不是 canonical dataset order。

B. Always source order  
Pros: 最 deterministic，跟 DS「優先消費既有 / source SSOT」最乾淨。Cons: 每次個別選取都重排，會違反 Atlassian standard example 所代表的 user selection order expectation。

C. Click order until Select All → source order reset  
Pros: 這就是 DS 現行 SelectMenu：個別選取由 `selectedValues` append；Select All 直接 `selectableOptions.map(o => o.value)` source reset。也符合「bulk action = consume option source as SSOT」。Cons: 跟 Ant Transfer 不完全一致。

我的 vote：C。不要再 claim「Ant Select 也是這樣」；正確說法是「世界級無統一共識，DS 選 C，因為 Select All 是 bulk reset 到 options source SSOT，個別點選保留 user intent」。
